# Real Usage Guide — Local LLM Security Engine

This guide explains how to use this service in practice: what data to send, how to read the response, what the fields actually mean, and how to make decisions based on the output.

---

## The typical request flow

1. A security event arrives at your SOC backend (from a SIEM, log pipeline, or alert system).
2. Your backend normalizes the event into the fields the engine expects.
3. Your backend sends a `POST /analyze-event` request to the engine.
4. The engine returns a structured `AnalysisResponse`.
5. Your backend checks `fallback_used` **first**.
6. If `fallback_used: false`, the classification is model-produced and usable for routing.
7. If `fallback_used: true`, the result is a safe default — route for manual review.

The engine's job ends at step 4. What you do with the result is entirely up to your SOC backend.

---

## What to send

### The description field

This is the only required field. It should be a normalized, human-readable summary of the event — not a raw log line, but not so vague that the model has nothing to work with.

**Good descriptions:**
- `"57 failed SSH logins from 185.220.101.1 followed by one successful login on port 22."`
- `"Outbound DNS query to C2 domain evil-server.xyz from internal host 10.0.0.45."`
- `"LSASS memory access from non-system process mimikatz.exe on DC01."`
- `"Scheduled task created to run base64-encoded PowerShell at startup."`

**Weak descriptions (the model may return `unknown` or a poor classification):**
- `"auth_failure"` — too terse; no context
- `"Alert generated by rule 2010935"` — no information about the event itself
- `"See attached logs"` — no content at all

The model makes better decisions with more context. If you have IPs, timestamps, event types, and count information, include them.

### Optional fields

| Field | When to include |
|---|---|
| `source_ip` | Any network event |
| `destination_ip` | Outbound/lateral movement events |
| `event_type` | When you have a normalized type (e.g. `"authentication_failure"`, `"port_scan"`, `"dns_lookup"`) |
| `severity` | When your SIEM already scored the event |
| `timestamp` | Always, if available — helps with context-aware reasoning |
| `additional_context` | Threat intel hits, account context, prior activity, enrichment |

### What NOT to send

- **Raw log lines**: Syslog, JSON event arrays, XML — the model is not a log parser. Pre-process into a human-readable description first.
- **PII beyond what is necessary**: The description goes directly into the model prompt. Minimize personal data.
- **Very long free text**: The description is capped at 4,000 characters. Very long inputs hit the limit and return a 422 error before reaching the model.
- **Sensitive secrets**: Do not put passwords, API keys, or credentials in the description even if they appear in the original log.

## How to read the response

### The most important field: `fallback_used`

Check this **before** looking at any other field.

```json
"fallback_used": false
```

`false` means: the model produced output, the parser extracted valid JSON, and the validator confirmed every field was in range. The classification is model-produced.

```json
"fallback_used": true
```

`true` means: something went wrong. The model output could not be parsed, or a field was out of range, or Ollama failed entirely. The values in `attack_classification`, `risk_score`, and `false_positive_likelihood` are **safe defaults, not real classifications**:

```json
{
  "attack_classification": "unknown",
  "false_positive_likelihood": 0.5,
  "risk_score": 50,
  "reason": "Analysis failed or returned invalid data. Manual review required.",
  "fallback_used": true
}
```

When `fallback_used: true`, route the event for human review. Do not use the classification for automated decisions.

---

### `attack_classification` — what each label means

| Label | What it means | Example event |
|---|---|---|
| `reconnaissance` | Scanning, probing, information gathering — **before** any access is gained | Nmap scan of external firewall, failed port probe |
| `credential_access` | Trying to steal or brute-force credentials — **access not yet gained** | 57 failed SSH logins, password spray, LSASS dump |
| `initial_access` | **First successful entry** into an environment | Phishing link clicked + shell, brute-force followed by login success |
| `lateral_movement` | Moving between systems **inside** an already-compromised network | Admin share access from a workstation, Kerberoasting |
| `command_and_control` | Outbound communication to attacker infrastructure from a compromised host | Beacon to C2 IP, DNS tunneling |
| `benign` | Normal, expected activity — no realistic threat scenario | Scheduled backup job, expected admin login from known location |
| `unknown` | Cannot classify with confidence from available data | Ambiguous event with minimal context |

**Common confusion: `credential_access` vs `initial_access`**

- 57 failed logins → `credential_access` (still trying to get in)
- 57 failed logins + 1 success → `initial_access` (access was gained)
- Password hash dump from memory → `credential_access`
- Attacker logging in with stolen credentials → `initial_access`

The model has explicit disambiguation examples in its prompt, so it handles this well when the description contains enough detail.

---

### `risk_score` — how to interpret it

`risk_score` is an integer from 0 to 100. Think of it as the model's estimate of how urgently the event needs attention:

| Range | What it suggests |
|---|---|
| 0–20 | Low concern — benign or very low confidence of threat |
| 21–50 | Worth noting — ambiguous or low-severity activity |
| 51–75 | Elevated — investigate soon |
| 76–89 | High — investigate promptly |
| 90–100 | Critical — investigate immediately |

Do not use `risk_score` as a hard threshold without calibrating it against your own data. A score of 70 on a well-described lateral movement event may deserve more attention than a score of 80 on an ambiguous event with a vague description.

---

### `false_positive_likelihood` — how to use it

This is a float from 0.0 to 1.0:

- `0.0` = the model is confident this is a real threat, not a false positive
- `1.0` = the model is confident this is a false positive
- `0.5` = the model is uncertain (also the fallback value when `fallback_used: true`)

Use it in combination with `risk_score`. A high `risk_score` combined with low `false_positive_likelihood` is a stronger signal than either field alone.

---

### `reason` — read this carefully

The `reason` field is a short explanation from the model about why it chose this classification. Read it before acting on the result:

```json
"reason": "57 failed SSH logins followed by a successful login from a flagged IP is consistent with credential brute-force leading to initial access."
```

A good reason references specific indicators from your input. A vague reason (like `"Suspicious activity detected"`) often indicates the model lacked enough context to reason confidently.

When `fallback_used: true`, the reason explains what went wrong:
```json
"reason": "Analysis failed or returned invalid data. Manual review required."
```

---

### Provenance fields — for debugging

| Field | What it tells you |
|---|---|
| `model_used` | Which Ollama model produced this result |
| `raw_parse_success` | Whether the raw model output was valid JSON (false = parser had to repair it) |
| `parse_strategy` | Which of the 6 extraction strategies succeeded (null if all failed) |
| `ollama_error` | The error message from Ollama if it failed (null on success) |
| `request_id` | Trace ID for correlating this response across logs |

If `fallback_used: true`, check `ollama_error` first. Common values:
- `"Ollama connection refused"` → Ollama is not running
- `"Ollama request timed out"` → model took too long to respond
- `"Model 'phi4-mini' not found"` → model was not pulled

## Example requests and responses

### Example 1 — Clear brute-force with success

**Request:**
```json
{
  "description": "57 failed SSH login attempts from 185.220.101.1 followed by one successful login. Source IP is on Spamhaus DROP list.",
  "source_ip": "185.220.101.1",
  "destination_ip": "10.0.0.22",
  "event_type": "authentication_failure",
  "severity": "high",
  "timestamp": "2024-01-15T10:00:00Z"
}
```

**Expected response** (`...` means additional response fields are omitted for brevity — the real response includes all fields listed in the Response Schema section above):
```json
{
  "attack_classification": "credential_access",
  "false_positive_likelihood": 0.05,
  "risk_score": 88,
  "reason": "High volume of failed SSH logins followed by success from a known-bad IP is consistent with credential brute-force.",
  "fallback_used": false,
  ...
}
```

Note: The model may classify this as `initial_access` if the description emphasizes the successful login. Either is defensible — `credential_access` emphasizes the attack technique, `initial_access` emphasizes that access was gained.

**Expected response:**

### Example 3 — Ambiguous event (insufficient context)

**Request:**
```json
{
  "description": "User logged in from an unusual location.",
  "event_type": "login",
  "severity": "medium"
}
```

**Expected response:**
```json
{
  "attack_classification": "unknown",
  "false_positive_likelihood": 0.5,
  "risk_score": 40,
  "reason": "Insufficient context to classify. 'Unusual location' without IP, user history, or geographic details cannot be reliably classified.",
  "fallback_used": false,
  ...
}
```

This is the correct behavior for an ambiguous event. The model returns `unknown` rather than guessing. Add more context to get a better classification.

## Using `request_id` for debugging

Always pass an `X-Request-ID` header when calling the engine from a SOC backend. Use the same ID throughout the call chain:

```bash
curl -X POST http://localhost:8000/analyze-event \
  -H "X-Request-ID: siem-alert-id-12345" \
  -H "Content-Type: application/json" \
  -d '{"description": "..."}'
```

The engine echoes this ID in:
- The `X-Request-ID` response header
- The `request_id` field in the response body
- Every structured log line for this request

When a request fails or returns unexpected results, the `request_id` lets you find the exact log entries for that request on both the engine and the SOC backend side.

## Consuming the stable response schema safely

The `AnalysisResponse` schema is stable and all fields are always present. Safe integration pattern:

```python
result = response.json()

# ALWAYS check this first
if result["fallback_used"]:
    log.warning("LLM fallback", extra={"request_id": result["request_id"], "ollama_error": result["ollama_error"]})
    route_for_manual_review(result)  # send to analyst queue; do not use the classification
    return

# Now safe to use the classification
classification = result["attack_classification"]  # one of 7 known values
risk_score = result["risk_score"]                  # int in [0, 100]
fp_likelihood = result["false_positive_likelihood"] # float in [0.0, 1.0]

# The reason is always a non-empty string — good for alert notes
note = result["reason"]
```

You can always depend on:
- `attack_classification` is one of the 7 allowed string values
- `false_positive_likelihood` is a float in [0.0, 1.0]
- `risk_score` is an int in [0, 100]
- `reason` is a non-empty string
- `fallback_used` is a boolean

These invariants hold across all code paths — whether Ollama succeeded, failed, timed out, or returned unparseable output.
